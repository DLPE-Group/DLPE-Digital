# S0 — Tenant Isolation Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make every tenant-scoped row carry a non-null `tenantId` and enforce hard isolation in Postgres (RLS), so tenant A can never read or write tenant B's data — the prerequisite for any multi-customer SaaS.

**Architecture:** Additive, expand/contract rollout (per `docs/DEPLOYMENT-RULES.md`): add a `Tenant` table and nullable `tenantId` to every tenant-scoped model, backfill all existing rows to one "DLPE demo" tenant, then flip `tenantId` to NOT NULL. Requests connect as a non-superuser role (`il_app`) and run inside a transaction that sets `SET LOCAL app.tenant`; Postgres RLS policies filter every table by `tenantId = current_setting('app.tenant')`. Migrations and seed/provisioning keep connecting as the table-owning superuser (`postgres`), which bypasses RLS by design.

**Tech Stack:** Express + Prisma 6.19.3 + Postgres 16 (Docker `il_postgres` locally, DigitalOcean Managed Postgres in prod), Vitest API suite (port 4100, `NODE_ENV=test`).

## Global Constraints

- **Subsystem scope:** This is S0 only (tenant foundation). The provisioning engine (S1) is a separate plan. Do not build blueprints, `provisionTenant`, or platform APIs here.
- **A tenant = the top-level `GROUP` `OrgNode`.** `Tenant.id` is its own id; a tenant's org root is its `GROUP` node. The existing `loadTenantResolver` (`server/src/domain/tenancy.ts`) already maps any `companyId` → its GROUP id.
- **`CountryDefaults` stays GLOBAL** — it is shared reference data and gets **no** `tenantId` and **no** RLS. (Confirmed judgment call.)
- **Expand/contract migrations only** (`docs/DEPLOYMENT-RULES.md` rule 2): never break the running code version. Add nullable → backfill → set NOT NULL, each a separate migration. Never `prisma migrate dev` against shared/test DBs in CI; author SQL + `prisma migrate deploy`. `migrate reset` is blocked for AI agents.
- **Migrations run as `postgres` (table owner / superuser); the app connects as `il_app` (non-superuser, no `BYPASSRLS`).** Two connection strings: `DATABASE_URL` (owner, used by migrate/seed/provisioning) and `APP_DATABASE_URL` (il_app, used to serve requests).
- **Commit trailer:** end every commit message with `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
- **Tenant-scoped models (26)** — every one gets `tenantId` + RLS: `OrgNode, User, UserScope, Session, Role, FieldRule, RbacVersion, StageConfig, CrossTrigger, FleetOperator, Invoice, VehicleTimeline, TimelineEvent, Integration, AuditEntry, AuditCascade, Report, DashboardLayout, DataSharing, UserPreference, PortalMessage, TrackDef, StageDef, EntityType, FieldDef, Entity`. **NOT scoped:** `CountryDefaults` (global), `_prisma_migrations` (Prisma-internal).

- [ ] **Step 3: Author the role migration**

Create `server/prisma/migrations/<ts>_app_role/migration.sql`:

```sql
-- Application role: LOGIN, NOT a superuser, NO BYPASSRLS, owns no tables.
DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'il_app') THEN
    CREATE ROLE il_app LOGIN PASSWORD 'il_app_pw';
  END IF;
END $$;

GRANT USAGE ON SCHEMA public TO il_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO il_app;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO il_app;
-- Future tables created by the owner are auto-granted to il_app:
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO il_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT USAGE, SELECT ON SEQUENCES TO il_app;
```

```ts
import { PrismaClient } from '@prisma/client';
import { appDatabaseUrl } from '../env.js';

// Dedicated client that connects as the RLS-enforced il_app role.
export const appPrisma = new PrismaClient({
  datasources: { db: { url: appDatabaseUrl } },

// Run `fn` inside a transaction scoped to one tenant. SET LOCAL keeps the GUC
// bound to this transaction only, so pooled connections never leak context.
export async function withTenant<T>(
  tenantId: string,
  fn: (tx: Parameters<Parameters<typeof appPrisma.$transaction>[0]>[0]) => Promise<T>,
): Promise<T> {
  return appPrisma.$transaction(async (tx) => {
    // set_config(name, value, is_local=true) — parameterized, injection-safe.
    await tx.$executeRaw`SELECT set_config('app.tenant', ${tenantId}, true)`;
    return fn(tx);
  });
}
```
